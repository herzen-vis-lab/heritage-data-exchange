# Heritage Data Exchange
 
> **Prototype of a federated exchange protocol for digital cultural heritage objects between universities, museums, and libraries in a loosely coupled network.**
 
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Protocol Version](https://img.shields.io/badge/protocol-v0.0.4-green.svg)](docs/federated-exchange-document.md)
 
---
 
Разрабатывается в РГПУ им. А. И. Герцена
 
Прототип федеративного MDM для слабосвязанной сети независимых организаций —
реализован на домене культурного наследия, архитектурные паттерны применимы к холдинговым структурам
и межведомственному обмену в КИИ. Решает три задачи: идентификация узлов без центрального оператора,
связи объектов между независимыми хранилищами, согласование НСИ в федеративной топологии.
 
---
 
## Проблема / Problem
 
Существующие форматы описания культурного наследия (MARC21, Dublin Core, IIIF, EDM) создавались для **централизованных** каталогов. При переходе к децентрализованной сети независимых организаций они не решают три ключевых задачи:
 
- **Идентификация узлов сети** — нет механизма уникальной идентификации организации-источника при обмене данными между узлами слабосвязанной сети
- **Связывание записей между узлами сети** — нет стандартного протокола обнаружения дубликатов и взаимного обогащения записей
- **Совместимость НСИ** — классификаторы и словари локальны, нет механизма согласования авторитетных источников между узлами сети
Проект исследует **протокол обмена для децентрализованной слабосвязанной сети** как альтернативу единому централизованному каталогу.
 
---
 
## Архитектурный подход
 
Проект применяет паттерны **федеративного MDM (Master Data Management)** к домену цифрового культурного наследия. Те же архитектурные паттерны применяются при построении межведомственного НСИ в холдинговых структурах и организациях КИИ.
 
| **MDM-паттерн** | **Реализация в проекте** |
|---|---|
| Идентификация мастер-записи | `digital_object_guid` (UUID v4) — уникальный идентификатор объекта в федерации |
| Связывание записей (Record Linkage) | Механизм обнаружения дубликатов между узлами + двунаправленное обогащение атрибутов |
| Классификация через авторитетные источники | `Classification.authority` (URI) — внешние словари (AAT, ULAN, LCSH, VIAF) |
| Федеративная топология | Каждый узел хранит данные локально; центральная система не требуется |
| Стюардство данных | Атрибутирование с верификацией на уровне эксперта (`attribution_status`) |
 
**Почему федеративная топология, а не реестровая / звезда:**
организационная автономность участников и отсутствие доверенного центрального оператора — стандартные условия как для музейных консорциумов, так и для холдингов с независимыми юрлицами.

## Контекст и публикации
 
**Область применения:** университеты, музеи, библиотеки и прочие организации с накопленным культурным наследием и потребностью в совместном цифровом каталоге без централизованного оператора.
 
**Смежные кейсы из enterprise-практики:** задача идентична построению федеративного НСИ (MDM) для холдинговых структур с независимыми юрлицами — та же проблема суверенитета данных, тот же выбор топологии, те же паттерны организации взаимосвязи между узлами сети.
