# Heritage Data Exchange

> **Prototype of federated exchange of digital cultural heritage objects between universities, museums, libraries and other organizations in a loosely coupled network.**
> 
>   прототип федеративного MDM для слабосвязанной сети независимых организаций —
>  реализован на домене культурного наследия, архитектурные паттерны применимы к холдинговым структурам
>  и межведомственному обмену в КИИ. Решает три задачи: идентификация узлов без центрального оператора,
>  связи объектов между независимыми хранилищами, согласование НСИ в федеративной топологии.

Разрабатывается в РГПУ им. А. И. Герцена.

---

## Проблема / Problem

Существующие форматы описания культурного наследия (MARC21, Dublin Core, IIIF, EDM) создавались для **централизованных** каталогов. При переходе к децентрализованной сети независимых организаций (слабосвязанной сети) они не решают три ключевых задачи:

- **Идентификация узлов сети** — нет механизма уникальной идентификации организации-источника при обмене данными  между узлами слабосвязанной сети
- **Связывание записей между узлами сети** — нет стандартного протокола обнаружения дубликатов и взаимного обогащения записей между узлами слабосвязанной сети
- **Совместимость НСИ** — классификаторы и словари локальны, нет механизма согласования авторитетных источников между узлами слабосвязанной сети

Проект исследует **протокол обмена для децентрализованной слабосвязанной сети** как альтернативу единому централизованному каталогу.

---

## Архитектурный подход

Проект применяет паттерны **Федеративной схемы ведения НСИ (MDM)** к домену цифрового культурного наследия:

| **НСИ-паттерн** | **Реализация в проекте** |
|---|---|
| Идентификация мастер-записи | `digital_object_guid` (UUID) — уникальный идентификатор объекта в федерации |
| Связывание записей | Механизм обнаружения дубликатов между узлами + двунаправленное обогащение атрибутов |
| Классификация | `Type.authority` (URI) — внешние авторитетные источники для классификации |
| Федеративная топология | Каждый узел хранит данные локально; нет центральной системы |
| Администраторы (стюарды) данных  | Верификация уровня знаний экспертов-атрибуторов |

Выбор **федеративной топологии** (против реестровой/схемы звезда) обусловлен требованиями организационной автономности участников и отсутствием доверенного центрального оператора.

## Контекст и публикации

**Область применения:** университеты, музеи, библиотеки и прочие организации с накопленным культурным наследием и потребностью в совместном цифровом каталоге без централизованного оператора.

**Смежные кейсы из enterprise-практики:** задача идентична построению федеративного НСИ (MDM) для холдинговых структур с независимыми юрлицами — та же проблема суверенитета данных, тот же выбор топологии, те же паттерны организации взаимосвязи элементов между узлами сети.
